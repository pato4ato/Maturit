1.2 Formati di Scambio Dati: JSON vs XML

Lo scambio di dati tra sistemi eterogenei avviene principalmente tramite due formati di serializzazione strutturata: XML (eXtensible Markup Language) e JSON (JavaScript Object Notation).

Caratteristica XML (eXtensible Markup Language) JSON (JavaScript Object Notation)
Struttura Gerarchica basata su tag annidati definiti dall'utente. Coppie chiave-valore basate sulla sintassi degli oggetti JS.
Trasparenza/Peso Verboso a causa dei tag di chiusura; richiede più banda. Leggero, essenziale, ottimizzato per il parsing web veloce.
Tipi di Dati Tratta tutto come testo; richiede validazione via XML Schema/DTD. Supporta nativamente stringhe, numeri, booleani, array, oggetti.
Parsing Richiede parser complessi come DOM (struttura ad albero) o SAX. Parser nativo integrato nei browser moderni (JSON.parse()).
Syntax/Code Block: Parsing XML in Ambiente Java

Esempio pratico di lettura e parsing di un file XML (Esercitazione Gestione Magazzino/Spesa) utilizzando l'approccio DOM (Document Object Model) incorporato nel JDK di Java:

import javax.xml.parsers.DocumentBuilderFactory;
import javax.xml.parsers.DocumentBuilder;
import org.w3c.dom.Document;
import org.w3c.dom.NodeList;
import org.w3c.dom.Node;
import org.w3c.dom.Element;
import java.io.File;

public class ParsingMagazzino {
    public static void main(String[] args) {
        try {
            File inputFile = new File("magazzino.xml");
            DocumentBuilderFactory dbFactory = DocumentBuilderFactory.newInstance();
            DocumentBuilder dBuilder = dbFactory.newDocumentBuilder();
            Document doc = dBuilder.parse(inputFile);
            doc.getDocumentElement().normalize();
            
            System.out.println("Elemento radice: " + doc.getDocumentElement().getNodeName());
            NodeList nList = doc.getElementsByTagName("articolo");
            
            for (int temp = 0; temp < nList.getLength(); temp++) {
                Node nNode = nList.item(temp);
                if (nNode.getNodeType() == Node.ELEMENT_NODE) {
                    Element eElement = (Element) nNode;
                    System.out.println("ID Articolo: " + eElement.getAttribute("id"));
                    System.out.println("Nome: " + eElement.getElementsByTagName("nome").item(0).getTextContent());
                    System.out.println("Quantità: " + eElement.getElementsByTagName("quantita").item(0).getTextContent());
                }
            }
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

1.3 Evoluzione delle Architetture Web: Dai Monoliti ai Microservizi

Le architetture web classiche di tipo Monolitico concentrano la logica di presentazione, di business e l'accesso ai dati all'interno di un unico perimetro applicativo ed esecutivo. Sebbene semplici da sviluppare inizialmente, i monoliti soffrono di rigidezza strutturale, scalabilità inefficiente (è necessario scalare l'intera applicazione anche se una sola risorsa è satura) e single point of failure (un bug in un modulo può arrestare l'intero server).

L'architettura a Microservizi scompone l'applicazione in servizi autonomi, circoscritti a un singolo dominio di business. Ogni microservizio comunica con gli altri tramite protocolli leggeri (tipicamente API REST via HTTP/JSON) e dispone di un proprio database indipendente.

Concetti Chiave: Containerizzazione e Orchestrazione
  • Docker: Tecnologia di containerizzazione che permette di impacchettare un microservizio insieme a tutte le sue dipendenze, librerie e file di configurazione in un "container" isolato, garantendo l'immutabilità e l'identità di comportamento tra ambienti di sviluppo, test e produzione.
  • Kubernetes (K8s): Piattaforma di orchestrazione open-source creata per automatizzare il deployment, lo scaling, il bilanciamento del carico e la guarigione automatica (self-healing) di cluster complessi formati da migliaia di container Docker.
Collegamenti Interdisciplinari: Calvino e la Combinatoria dei Sistemi

Informatica - UdA 2: Basi di Dati e Progettazione Concettuale e Logica

2.1 Introduzione ai Database e Architettura a Tre Livelli

Un Database è una collezione organizzata di dati strutturati, gestita in modo centralizzato da un software specifico chiamato DBMS (Database Management System). Il DBMS garantisce l'efficienza nell'accesso, la consistenza, l'affidabilità dei dati e il controllo degli accessi concorrenti. L'architettura standard ANSI-SPARC definisce tre livelli distinti di astrazione per separare le applicazioni utente dall'effettiva memorizzazione fisica:

  • Livello Esterno (Viste): Rappresenta la porzione di database visibile a un utente o a uno specifico sottogruppo applicativo. Nasconde le strutture non pertinenti.
  • Livello Logico: Descrive l'intera struttura logica globale del database in termini di tabelle, vincoli e relazioni (indipendentemente dall'hardware e dal software di memorizzazione).
  • Livello Interno (Fisico): Definisce la reale memorizzazione fisica dei dati sui supporti di memoria di massa (strutture dei file, indici B-Tree, allocazione dei blocchi).

2.2 Sicurezza, Riservatezza e Integrità dei Dati

Il DBMS deve operare come guardiano stringente per assicurare la Privatezza (attraverso meccanismi di autenticazione e autorizzazione), l'Integrità Fisica (tramite log delle transazioni e procedure di backup/recovery in caso di guasto hardware) e la Consistenza (impedendo che transazioni concorrenti interferiscano distruggendo la coerenza logica dello stato dei dati).

2.3 Progettazione Concettuale: Il Modello Entità-Relazione (E/R)

La progettazione di una base di dati si articola in tre grandi fasi sequenziali: Progettazione Concettuale, Progettazione Logica e Progettazione Fisica. Il Modello Entità-Relazione è lo strumento principe della fase concettuale, traducendo i requisiti della realtà in uno schema grafico indipendente dalla tecnologia finale.

Costrutti del Modello E/R
  • Entità: Classi di oggetti (es. Studente, Libro) del mondo reale che hanno un'esistenza autonoma e rilevanza per l'applicazione.
  • Attributi: Proprietà elementari che descrivono le entità o le associazioni (es. CodiceFiscale, Titolo).
  • Chiave Primaria (Identificatore): Un attributo o un insieme di attributi che identificano univocamente una specifica istanza dell'entità.
  • Associazioni (Relazioni): Legami logici stabili tra due o più entità.
  • Cardinalità: Specificano il numero minimo e massimo di istanze di un'entità che possono essere coinvolte nell'associazione (es. $(0,1)$, $(1,N)$, $(1,1)$).

2.4 Progettazione Logica: Derivazione al Modello Relazionale

La progettazione logica converte lo schema E/R in uno schema relazionale, basato su tabelle composte da righe (tuple) e colonne (attributi). Le regole di derivazione standard stabiliscono che:

  1. Ogni entità diventa una tabella (relazione).
  2. Gli attributi dell'entità diventano le colonne della tabella.
  3. Le relazioni 1:N si convertono esportando la chiave primaria dell'entità "lato 1" all'interno della tabella dell'entità "lato N", dove assume il ruolo di Chiave Esterna (Foreign Key) per implementare l'integrità referenziale.
  4. Le relazioni N:N generano una nuova tabella ponte intermedia, avente come chiave primaria composta l'unione delle chiavi primarie delle due entità originarie, configurate anche singolarmente come chiavi esterne.

2.5 Normalizzazione delle Basi di Dati

La normalizzazione è un procedimento formale di scomposizione delle tabelle volto ad eliminare la ridondanza dei dati e a prevenire le pericolose anomalie di inserimento, cancellazione e aggiornamento. Si basa sullo studio delle dipendenze funzionali.

Forme Normali (1FN, 2FN, 3FN)
  • Prima Forma Normale (1FN): Una tabella è in 1FN se e solo se tutti i suoi attributi sono atomici (non scomponibili e non multi-valore) e se esiste una chiave primaria definita.
  • Seconda Forma Normale (2FN): Una tabella è in 2FN se è in 1FN e tutti gli attributi non-chiave dipendono in modo completo dall'intera chiave primaria (non da una sua parte comune o sottoinsieme). È sempre verificata se la chiave primaria è semplice (singolo attributo).
  • Terza Forma Normale (3FN): Una tabella è in 3FN se è in 2FN e nessun attributo non-chiave dipende in modo transitivo dalla chiave primaria (ovvero, gli attributi non devono dipendere da altri attributi non-chiave).

Inglese - CLIL Module: Advanced Database Systems & Paradigm Shift

Relational vs Non-Relational Data Management

In contemporary software engineering, information management is divided into two primary storage architectural philosophies: Relational Systems (RDBMS) and Non-Relational Systems (commonly known as NoSQL). Traditional database engines process data using strictly defined algebraic constraints, enforcing transaction reliability. Conversely, modern large-scale applications require horizontally scalable paradigms to manage unstructured or semi-structured data matrices.

ACID Properties Atomicity, Consistency, Isolation, Durability; guarantees transactional integrity. "Financial platforms require absolute compliance with ACID properties."

Informatica - UdA 3: Algebra Relazionale e Interrogazione Avanzata (SQL)

3.1 Operatori dell'Algebra Relazionale

L'algebra relazionale è un linguaggio procedurale teorico basato su relazioni matematiche. Gli operatori primitivi fondamentali comprendono la Selezione ($\sigma$), che filtra le tuple orizzontalmente in base a una condizione; la Proiezione ($\pi$), che isola verticalmente un sottoinsieme delle colonne eliminando i duplicati; il Prodotto Cartesiano ($\times$), che accoppia ogni riga della prima tabella con ogni riga della seconda; e la Congiunzione o Inner Join ($\bowtie$), che unisce le righe correlate da una corrispondenza logica esplicita tra attributi comuni.

3.2 Tipologie di JOIN in SQL

In SQL standard (implementato ed esercitato su PhpMyAdmin/MySQL), il collegamento tra tabelle si realizza mediante costrutti semantici precisi:

  • INNER JOIN: Restituisce unicamente le righe che trovano esatta corrispondenza in entrambe le tabelle coinvolte nella condizione di join.
  • LEFT OUTER JOIN: Restituisce tutte le righe della tabella sinistra, accoppiate con quelle della tabella destra se disponibili; se non vi è corrispondenza, inserisce valori NULL nelle colonne della tabella destra.
  • RIGHT OUTER JOIN: Speculare al precedente; preserva tutte le istanze della tabella destra riempiendo con NULL le colonne mancanti a sinistra.
  • FULL OUTER JOIN: Unisce i risultati di Left e Right Join, mantenendo tutte le tuple di entrambe le tabelle e inserendo NULL dove manca la corrispondenza (in MySQL si emula con la UNION di Left e Right Join).

3.3 Aggregazioni, Filtri Complessi e Sottoquery

La manipolazione avanzata richiede il raggruppamento e l'elaborazione statistica dei record tramite funzioni aggreganti (COUNT, SUM, AVG, MIN, MAX) pilotate dalla clausola GROUP BY.

La Distinzione Cruciale: WHERE vs HAVING
La clausola WHERE opera come filtro preventivo sulle singole righe prima che avvenga l'aggregazione dei dati. La clausola HAVING agisce invece come filtro post-aggregazione, applicandosi direttamente ai gruppi generati dal raggruppamento.
Syntax/Code Block: Query Avanzata con Sottoquery e Viste

Scenario reale: Selezione dei clienti che hanno effettuato ordini superiori alla media aziendale, con utilizzo di query annidate (IN, ALL), vincoli fisici di aggiornamento a cascata e creazione di una vista virtuale.

-- Creazione di una vista logica virtuale per motivi di sicurezza e riuso del codice
CREATE VIEW VistaOrdiniTop AS
SELECT id_cliente, COUNT(id_ordine) AS totale_ordini, SUM(importo_totale) AS spesa_complessiva
FROM ordini
WHERE data_ordine >= '2025-01-01'
GROUP BY id_cliente
HAVING SUM(importo_totale) > (
    -- Sottoquery scalare per determinare la media generale degli ordini aziendali
    SELECT AVG(importo_totale) 
    FROM ordini
);

-- Interrogazione complessa con costrutto IN e relazioni protette da vincoli fisici
SELECT c.id_cliente, c.cognome, c.nome, v.spesa_complessiva
FROM clienti c
INNER JOIN VistaOrdiniTop v ON c.id_cliente = v.id_cliente
WHERE c.id_cliente IN (
    SELECT DISTINCT id_cliente 
    FROM ordini 
    WHERE importo_totale >= ANY (SELECT importo_totale FROM ordini WHERE id_cliente = 101)
);

-- DCL (Data Control Language): Assegnazione e revoca dei privilegi di sicurezza sulla vista

Informatica - UdA 4: Programmazione Web Lato Server (PHP) e Sicurezza

4.1 Sintassi Fondamentale, Scope delle Variabili e Modularità

PHP (Hypertext Preprocessor) è un linguaggio di scripting interpretato lato server espressamente concepito per la generazione di pagine web dinamiche. Le variabili sono debolmente tipizzate e iniziano con il simbolo del dollaro ($). Lo scope delle variabili distingue nettamente tra variabili locali (visibili solo all'interno della funzione in cui sono dichiarate) e globali. Per iniettare costanti o librerie comuni salvaguardando il principio di riusabilità del codice DRY (Don't Repeat Yourself), si utilizzano le direttive strutturali:

  • include 'file.php'; Include il codice; in caso di errore logico o mancanza del file, emette un Warning e prosegue l'esecuzione.
  • require 'file.php'; Identico a include, ma in caso di errore blocca irreversibilmente l'esecuzione sollevando un Fatal Error. Da preferire per file di configurazione DB o script di controllo delle sessioni.

4.2 Interazione HTML-PHP e Sanitizzazione dell'Input

I dati inviati da form client-side vengono catturati sul server tramite array associativi globali predefiniti: $_GET (parametri visibili esplicitamente nella Query String dell'URL, ideale per ricerche e navigazione non sensibile) e $_POST (parametri incorporati nel corpo della richiesta HTTP, obbligatorio per password, transazioni e dati personali). La sicurezza impone il rifiuto categorico della fiducia negli input utente: i caratteri speciali devono essere neutralizzati tramite htmlspecialchars() per sventare attacchi di tipo XSS (Cross-Site Scripting).

Syntax/Code Block: Connessione Sicura PDO e Script di Login

Script PHP modulare e sicuro per l'interazione con database MySQL tramite l'estensione PDO (PHP Data Objects). Implementa le Prepared Statements per annullare totalmente il rischio di vulnerabilità di tipo SQL Injection.

<?php
// db_config.php - File di configurazione centralizzato
$dsn = "mysql:host=localhost;dbname=Hogwarts;charset=utf8mb4";
$db_user = "root";
$db_pass = "PasswordSicura2026!";
$options = [
    PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
    PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,
    PDO::ATTR_EMULATE_PREPARES   => false,
];

try {
    $pdo = new PDO($dsn, $db_user, $db_pass, $options);
} catch (\PDOException $e) {
    throw new \PDOException($e->getMessage(), (int)$e->getCode());
}

// login_process.php - Elaborazione autenticazione sicura
require_once 'db_config.php';
session_start();

if ($_SERVER["REQUEST_METHOD"] == "POST") {
    // Sanitizzazione dell'input per eliminare rischi di attacchi combinati
    $username = filter_input(INPUT_POST, 'username', FILTER_SANITIZE_SPECIAL_CHARS);
    $password = $_POST['password']; // Le password non si sanitizzano, si verificano gli hash

    if ($username && $password) {
        // Preparazione della query parametrizzata (Previene SQL Injection)
        $stmt = $pdo->prepare("SELECT id_utente, password_hash, ruolo FROM utenti WHERE username = :user");
        $stmt->execute(['user' => $username]);
        $user = $stmt->fetch();

        // Verifica crittografica della password tramite algoritmo nativo sicuro BCRYPT
        if ($user && password_verify($password, $user['password_hash'])) {
            // Rigenerazione dell'ID sessione per prevenire attacchi di Session Fixation
            session_regenerate_id(true);
            $_SESSION['user_id'] = $user['id_utente'];
            $_SESSION['ruolo'] = $user['ruolo'];
            $_SESSION['last_activity'] = time();
            
            header("Location: dashboard.php");
            exit();
        } else {
            $errore = "Credenziali non valide o inesistenti.";
        }
    }
}
?>

4.3 Persistenza dello Stato: Sessioni e Cookie

Il protocollo HTTP è intrinsecamente stateless (ogni transazione è indipendente). Per mantenere lo stato tra richieste successive si utilizzano due tecnologie distinte:

  • Cookie: Stringhe di testo memorizzate direttamente nel browser del client. Vengono inviati automaticamente al server ad ogni richiesta HTTP. Si impostano in PHP con la funzione setcookie(). Data la loro natura esposta sul client, sono soggetti a manipolazioni e sono vincolati dal rispetto della normativa europea GDPR (Cookie Banner) che impone il consenso esplicito prima del tracciamento dell'utente.
  • Sessioni: Meccanismo centralizzato in cui i dati sensibili sono memorizzati in file protetti sul server. Al client viene trasmesso esclusivamente un identificativo univoco effimero chiamato PHPSESSID memorizzato all'interno di un cookie tecnico di sessione. Si attivano invocando obbligatoriamente session_start().

Italo Svevo introduce nella cultura italiana la profonda rivoluzione della psicoanalisi freudiana e la crisi dell'uomo borghese dell'inizio del Novecento. I suoi personaggi non sono eroi romantici né operai del naturalismo, ma inetti: intellettuali e impiegati incapaci di adattarsi ai ritmi rigidi e competitivi della società capitalistica e industriale, paralizzati da una costante e ipertrofica autoanalisi intellettuale.

Analisi del Testo: "La coscienza di Zeno" (1923)

Il romanzo è strutturato come l'autobiografia del protagonista Zeno Cosini, redatta su invito del suo psicoanalista, il Dottor S., come parte della terapia. La narrazione non segue un asse temporale cronologico e oggettivo (dalla nascita alla vecchiaia), ma è organizzata per grandi nuclei tematici (Il fumo, La morte di mio padre, La storia del mio matrimonio, La moglie e l'amante).

Il Tempo Misto: Gli eventi si sovrappongono; il presente del narratore che scrive (vecchio e disincantato) si intreccia continuamente con il passato del protagonista che agisce (giovane e goffo). Zeno si rivela un narratore inattendibile: le sue memorie non sono resoconti storici esatti, ma continui tentativi di auto-giustificazione razionale dei propri fallimenti e delle proprie nevrosi. Il paradosso finale del romanzo mostra Zeno che si dichiara guarito non grazie alla psicoanalisi (da lui abbandonata), ma grazie ai successi commerciali ottenuti durante la speculazione bellica della Prima Guerra Mondiale, giungendo alla conclusione apocalittica che la vita stessa è "inquinata alle radici" e che solo una catastrofe cosmica potrà risanarla eliminando l'uomo dalla terra.

Informatica - UdA 5: Tecnologie Asincrone (AJAX) e Moduli CLIL

5.1 Architettura AJAX e Chiamate Asincrone

L'architettura AJAX (Asynchronous JavaScript and XML) rappresenta una svolta fondamentale nel web design e nello sviluppo applicativo. Consente alle pagine web di scambiare dati con il server in background, in modo completamente asincrono, aggiornando porzioni mirate e isolate dell'interfaccia grafica (DOM) senza richiedere il ricaricamento totale della pagina. Questo paradigma si implementa oggi tramite l'API moderna fetch() basata sulle Promises di JavaScript.

5.2 Programmazione Orientata agli Oggetti (OOP) in PHP vs Java

PHP supporta pienamente il paradigma Object-Oriented a partire dalla versione 5. I concetti chiave includono la definizione di Classi (Class), Oggetti (Object), Costruttori (__construct), Ereditarietà (Inheritance tramite la parola chiave extends), Incapsulamento (Encapsulation regolata dai modificatori public, protected, private) e Polimorfismo (Polymorphism).

Collegamenti Interdisciplinari: Joyce e lo Stream of Consciousness Asincrono
La natura non lineare, asincrona e interruttiva delle chiamate AJAX, che spezzano la sequenzialità classica del caricamento web sincrono, trova una corrispondenza metodologica formale nella letteratura modernista di lingua inglese di James Joyce. Lo Stream of Consciousness (Flusso di Coscienza) rompe la sintassi tradizionale e la punteggiatura del romanzo ottocentesco per riprodurre il fluire caotico, simultaneo e asincrono dei pensieri e delle impressioni sensorie della mente umana, operando un vero e proprio "aggiornamento parziale" continuo della coscienza del personaggio senza una regia narrativa centralizzata.

Inglese - James Joyce: Modernism and the Stream of Consciousness

The Modernist Revolution and the Subversion of Narrative Time

At the beginning of the 20th century, the traditional Victorian novel collapsed under the weight of historical trauma (World War I) and revolutionary intellectual discoveries (Einstein's relativity, Freud's psychoanalysis, and Bergson's internal time). James Joyce stands as the most radical innovator of English literature, orchestrating a complete shift from external objective description to internal psychological exploration.

Textual Analysis: "Eveline" (from Dubliners, 1914)

Plot and Theme: Eveline is a nineteen-year-old girl trapped in a bleak, abusive Dublin household. She has the concrete opportunity to escape to Buenos Aires with her lover, Frank. The entire narrative takes place in a compressed external timeframe while she sits at the window, but expands immensely within her mind as she balances her duty to her dead mother against her desire for personal freedom.

The Epiphany: The climax represents a classic Joycean *epiphany* (a sudden spiritual manifestation triggered by a casual object or sound, in this case, a street organ playing). The music reminds her of her mother's miserable death. However, at the station, paralyzed by an unbearable, psychological inability to act (the core theme of *Dublin Paralysis*), she freezes and refuses to board the ship, looking at Frank like a helpless animal, completely incapable of altering her tragic destiny.

Textual Analysis: "Ulysses" (1922) and the Interior Monologue

Set entirely on a single day (June 16, 1904), *Ulysses* follows Leopold Bloom, Stephen Dedalus, and Molly Bloom through Dublin. Joyce completely discards traditional syntax, punctuation, and logical transitions in the final chapter (Molly's Monologue) to capture the unmediated Stream of Consciousness. Thoughts, memories, and physical sensations flow together asynchronously, creating a rich, multi-layered cognitive network that mirrors the decentralized complexity of a modern digital system.

Informatica - UdA 6: Simulazioni, Orientamento ed Esame di Stato

6.1 Analisi Metodologica della Seconda Prova d'Esame

La scomposizione di un tema d'esame ministeriale (es. i casi di studio storici "Terminal Marittimo" o "SportOvunque") impone un approccio ingegneristico strutturato rigoroso diviso in macro-fasi sequenziali:

  1. Analisi del Testo ed Enucleazione dei Requisiti: Identificazione delle entità coinvolte, delle regole di business e dei vincoli fisici descritti esplicitamente o implicitamente nella traccia.
  2. Progettazione Concettuale (Schema E/R): Tracciamento del diagramma formale definendo le cardinalità minime e massime e isolando le chiavi primarie identificative.
  3. Progettazione Logica (Schema Relazionale): Applicazione rigorosa delle regole di conversione in tabelle, evidenziando le chiavi esterne e i vincoli di integrità referenziale (ON UPDATE CASCADE, ON DELETE RESTRICT).
  4. Scrittura delle Interrogazioni (SQL): Sviluppo delle query di estrazione statistiche richieste dalla traccia tramite JOIN e funzioni aggreganti avanzate.
  5. Codifica Lato Server (PHP/PDO): Implementazione del prototipo applicativo di sicurezza (es. form di inserimento dati o autenticazione) adottando le Prepared Statements.
